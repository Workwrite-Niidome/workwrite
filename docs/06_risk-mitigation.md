# 超読者ファースト リスクと緩和策

## ビジネスリスク

| リスク | 影響度 | 発生確率 | 緩和策 |
|--------|--------|----------|--------|
| コールドスタート（初期データ不足でレコメンド精度低） | 最大 | 高 | オンボーディング診断で初期データ取得。マクロ・ミドル層で補完。キュレーション型おすすめ併用 |
| レビュー文化が根付かない | 大 | 高 | ポイントインセンティブ。3秒タグ形式に簡素化。ゲーミフィケーション |
| AIスコアリングの精度・公平性への疑念 | 大 | 中 | スコア算出ロジックの透明性確保。異議申立てフロー。人間レビューとのハイブリッド |
| 作家の抵抗感（「数値で作品を判断するな」） | 中 | 高 | 「埋もれないためのツール」ポジショニング。スコア非公開オプション。改善提案の丁寧な設計 |
| 収益化の遅延 | 大 | 中 | 省察機能でプレミアムプランを差別化。投げ銭は無料ユーザーでも利用可能 |
| コンテンツ品質の不均一 | 中 | 中 | AIスコアによる品質下限設定。編集支援。コンテンツガイドライン |
| 法規制・著作権問題 | 大 | 低 | 利用規約の弁護士確認。AI分析の同意フロー。著作権保護ポリシー |

## 技術リスク

| リスク | 影響度 | 緩和策 |
|--------|--------|--------|
| LLM APIコスト増大 | 大 | スコアリングは投稿時のみ（リアルタイム不要）。キャッシュ活用。モデルサイズ最適化 |
| LLM APIのレート制限・障害 | 中 | キュー制御。リトライ機構。フォールバック（簡易スコア） |
| スケーラビリティの壁 | 大 | MVP段階からDB設計を意識。Phase 2でKafka/K8s導入 |
| 検索精度（日本語） | 中 | Meilisearchの日本語トークナイザ検証。Phase 2でElasticsearch移行を検討 |

## MVP段階の重点KPI

| KPI | 目標値 | 計測方法 |
|-----|--------|----------|
| ユーザー登録数 | 1,000人/月 | DB集計 |
| 月間アクティブ読者 | 500人 | ログイン+閲覧 |
| 平均読了率 | 40%以上 | ReadingProgress集計 |
| 感情タグ入力率 | 読了者の30%以上 | EmotionTag集計 |
| レビュー投稿率 | 読了者の10%以上 | Review集計 |
| プレミアム転換率 | 登録者の5% | Stripe集計 |
| 作家投稿数 | 50作品/月 | Work集計 |
