# Workwriteが取り組む「小説の構造化」── AIが物語を理解するために、私たちが設計したこと

## 私たちが解こうとしている問題

Workwriteは「いい作品が生まれ、埋もれず、読者に届く」ための小説プラットフォームです。

その中核機能のひとつに**AIによる執筆アシスト**があります。キャラクターの口調を活かした会話の提案、プロットに沿った続きの執筆、世界観を踏まえたシーン描写の強化。

でも、このAI執筆アシストの開発で、私たちはずっとある問題と闘ってきました。

**「AIが、物語の中で起きていないことを、起きたかのように書いてしまう」**

たとえば、主人公がまだ誰とも約束していないのに「あの時の約束を胸に」と書く。前の話で怒って部屋を出たキャラクターが、次の話の冒頭で何事もなかったかのように笑っている。中世ファンタジーの世界で、突然スマートフォンが出てくる。

これはAIの知識が足りないのではなく、**AIに物語の構造が伝わっていないこと**が原因でした。

---

## 「全文を渡せばいい」は間違いだった

最初の私たちのアプローチは単純でした。「全話のテキストをAIに渡せば、AIは物語を理解して正しい続きを書けるだろう」と。

結論から言うと、これはうまくいきませんでした。

10話分の小説テキストを丸ごとAIに渡しても、AIは**「どの情報が今の執筆に重要なのか」を判断できません**。3話目で張った伏線も、7話目のモブの会話も、AIにとっては等しく「テキスト」です。

人間の作家は、頭の中に物語の構造を持っています。キャラクターの関係性、未回収の伏線、前の場面の雰囲気、今回の話で達成したいこと——これらを無意識に参照しながら書いている。

**AIにも、同じ「構造」を明示的に渡す必要がある。** 私たちの取り組みは、ここから始まりました。

---

## Workwriteが構造化している7つのレイヤー

私たちは、小説の裏側にある構造を7つのレイヤーに分解し、AIが「物語を理解している」状態を作っています。

### 1. キャラクター構造

キャラクターの名前や見た目だけではありません。Workwriteが管理しているのは：

- **性格・口調・一人称**（「僕」なのか「俺」なのか。丁寧語なのかぶっきらぼうなのか）
- **現在の状況**（第5話時点で、このキャラは何を知っていて、どんな感情状態にあるか）
- **関係性グラフ**（誰と誰が幼馴染で、誰と誰が敵対しているか）
- **代表的なセリフ例**（そのキャラクターらしい話し方の実例）

特に重要なのは**「現在の状況」の自動更新**です。キャラクターの状態は話が進むごとに変わります。第1話で明るかったキャラが、第5話では大切な人を失って沈んでいるかもしれない。Workwriteは各話を自動分析し、キャラクターの状態を常に最新に保ちます。

### 2. 世界設定の制約

ファンタジーなら「この世界に銃は存在しない」、SFなら「ワープ航法のルール」。世界設定は物語のルールブックです。

Workwriteでは世界設定を**制約として**AIに渡しています。「こういう技術が存在する」ではなく「こういう技術は存在しないから使ってはいけない」という制約です。

AIは新しいものを生み出すのは得意ですが、**「ないもの」を使わないことは苦手**です。だから、禁止事項として明示する。

### 3. プロット構造

「全体として、この物語はどこに向かっているのか」をAIに伝える層です。

三幕構成のどこにいるのか。中心的な葛藤は何か。この話は物語全体の中でどういう役割を持つのか。

プロット構造がなければ、AIは場当たり的に「面白そうな展開」を書きます。それは一話単位では良くても、物語全体の整合性を壊します。

### 4. 各話の要約と引き継ぎ

これが構造化の中でもっとも技術的に難しかった部分です。

各話の「要約」をAIに作らせると、**要約自体にハルシネーション（幻覚）が混入する**という問題が起きました。「主人公が検討した」が要約では「主人公が決意した」に変わり、次の話ではその「決意」が事実として扱われてしまう。

**嘘が要約に混じり、伝播し、やがて正史になってしまう。**

私たちはこれに対して、いくつかの対策を講じています：

- 要約プロンプトに**「推測禁止、原文の引用を必須」**という厳格なルールを設ける
- 要約だけでなく**前話の生テキスト末尾**も直接AIに渡す（要約を信じるのではなく、原文で接地する）
- 一定の話数ごとに**原文から要約を全再構築**し、蓄積した誤りをリセットする

### 5. 伏線管理

「第3話で張った伏線が、第12話で回収される」——これは人間の作家でも忘れがちなことです。

Workwriteは各話の分析時に伏線を自動検出し、「設置」「進展」「回収」の3状態で追跡します。AIが執筆アシストを行うとき、未回収の伏線リストが自動的にコンテキストに含まれます。

### 6. 感情設計

Workwriteには「感情ブループリント」という機能があります。この物語で読者にどんな感情の旅を届けたいかを、作家が設計できる仕組みです。

各話には「感情目標」を設定できます。「この話では読者に不安を感じさせ、最後に小さな希望を残す」といった設計意図を、AIアシストのコンテキストに含めることで、物語のトーンが一貫します。

### 7. 前話との「接続点」

最後に、もっとも地味だけど効果が大きかったのがこれです。

前話の要約ではなく、**前話の末尾テキストそのものを次の話の執筆コンテキストに含める**。

「夕日に溶けていく背中を、わたしはただ見つめていた」という一文の後に、AIが書き出す次の話の冒頭。これが要約経由では再現できない「場面の空気感の接続」を生みます。

## 構造化の全体像

整理すると、Workwriteの執筆アシストが参照する構造データはこうなります：

```
📖 AIが参照する構造コンテキスト
│
├── 🚫 ハルシネーション防止ルール（常に最優先）
├── 🎯 この話で達成すべきこと（章の目的・感情目標）
├── 📝 前話の末尾テキスト（生テキスト＝接地情報）
├── 🌍 世界設定と制約
├── 🎭 キャラクター設定・現在の状態・関係性
├── 📊 全体のプロット構造
├── 📋 これまでの話の要約
├── 🔗 未回収の伏線
├── 💭 感情設計（感情ブループリント）
└── ✍️ 文体情報（視点・トーン）
```

これらが**優先順位付きで、トークン予算内に収まるように**自動構成されます。すべての情報を毎回渡すのではなく、今の執筆に必要な情報を選別して渡す。ここにもノウハウがあります。

## 自分で試してみたい方へ

「仕組みはわかった。自分のChatGPTやClaudeでもやってみたい」という方のために、簡易版の方法を共有します。

### 最低限やるべき3つのこと

**① キャラクターシートを毎回プロンプトに含める**

```
【キャラクター設定（厳守）】

■ ルナ（主人公）
  一人称: わたし
  口調: 丁寧語ベース。親しい相手にはくだけた話し方
  性格: 好奇心旺盛だが臆病。正義感は強いが行動に移すまでに時間がかかる
  現在の状況: 第3話末時点で、カイトが禁止区域に行くと知り動揺している

■ カイト（幼馴染）
  一人称: 俺
  口調: ぶっきらぼう。「〜だろ」「〜じゃねえか」
  性格: 考えるより先に体が動くタイプ
  現在の状況: 明日禁止区域に行くと決めた。ルナには背を向けて去った
```

**② 前話の末尾（生テキスト）を必ず貼る**

要約ではなく、前話の最後の1000〜2000文字をそのままコピペしてください。

```
【前話（第3話）の末尾テキスト】
（ここに前話の最後の部分をそのまま貼り付け）
```

**③ 「やってはいけないこと」を明記する**

これが一番大事です。

```
【絶対禁止事項】
- 上記のテキスト・設定に存在しない過去の出来事を「あった」として書かないこと
- 「〇〇と約束した」「〇〇を思い出した」等は、提供テキストに根拠がある場合のみ
- キャラクターの一人称・口調を変えないこと
- この世界に存在しない現代技術を登場させないこと
```

### 統合プロンプト例

```
あなたは小説の執筆アシスタントです。
以下の設定と前話の情報に忠実に従い、続きを執筆してください。

【絶対禁止事項】
- 提供されたテキスト・設定に記述されていない過去のイベント、会話、約束を
  「あった」として書かないこと
- キャラクターの一人称・口調設定を絶対に変えないこと
- 世界設定に存在しない技術・概念を登場させないこと
- 不明な過去は参照せず、現在の場面に集中すること

【キャラクター設定（厳守）】
（ここにキャラクターシートを貼る）

【世界設定】
（ここに世界観の制約を貼る）

【前話の末尾テキスト】
（前話の最後1000-2000字をそのまま貼る）

【今回の話の目的】
- 何を達成する話か
- どんな感情を読者に届けたいか

【これまでの文章】
（今書いている途中の本文）

前話の末尾から自然に繋がる続きを、2000文字程度で書いてください。
```

これだけでも、何も構造化せずに「続きを書いて」と丸投げするよりは**格段にマシ**になります。

---

## ただし、これを毎話やるのは大変です

正直に言います。上の方法を毎話やるのは、**かなりの労力**です。

- キャラクターの「現在の状況」を毎話更新しないといけない
- 前話の引き継ぎ情報を毎回整理しないといけない
- 伏線を自分で追跡しないといけない
- 話数が増えるとプロンプトが肥大化して管理が破綻する
- AIに要約させると要約自体がハルシネーションする（前述の通り）

人力でやる構造化は、3〜5話までなら耐えられます。でも10話を超えたあたりから、**小説を書く時間より構造化データを整理する時間の方が長くなる**。

## Workwriteは、この全部を自動でやります

Workwriteでは、上で説明した7つのレイヤーの構造化を**システムとして自動化**しています。

- エピソードを保存するたびに自動分析が走り、キャラ状態・伏線・世界設定が更新される
- 執筆アシストを使うとき、必要な構造データが自動でAIに渡される
- 要約のハルシネーションを防ぐ仕組みが組み込まれている
- キャラクター設計、プロット設計、世界観構築もAIがサポート

**作家がやることは「書く」だけ。** 構造化はWorkwriteに任せて、創作に集中できます。

さらにWorkwriteには、AI品質スコアリング（6軸評価）、感情タグによる作品発見、ファンレター機能など、「いい作品が正しく評価され、読者に届く」ための仕組みがあります。
