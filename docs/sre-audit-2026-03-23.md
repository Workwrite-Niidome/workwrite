### 本番構成

- フロントエンド: Vercel (`workwrite.jp`)
- バックエンド: Railway (`backend-production-db434.up.railway.app`)
- API URL: `NEXT_PUBLIC_API_URL` でビルド時にフロントに埋め込み
- デプロイ: RailwayはCI成否に関係なく自動デプロイ

### 発見した問題

#### GET /works?limit=N が500を返す (MEDIUM)

- `PaginationDto` をintersection typeで使っている唯一のエンドポイント
- `@Query() query: PaginationDto & { genre?: string }` — NestJSはruntime type metadataを正しく処理できない可能性
- フロントエンドは `limit` パラメータなしで呼んでいる模様（limitなしなら200が返る）
- **ファイル**: `apps/backend/src/works/works.controller.ts:17`

#### /search エンドポイントが404 (LOW)

- SearchModuleのルーティングが未登録、またはコントローラ未実装
- フロントはdiscover経由で検索している模様

## Phase 5: 運用改善提案

### 現状の運用成熟度

| 領域 | カバー率 | 現状 |
|---|---|---|
| モニタリング | 20% | ヘルスチェック1本のみ。APM/メトリクス/ダッシュボードなし |
| ログ | 30% | 例外ログあり。Pinoは依存追加済みだが未統合。構造化ログなし |
| アラート | 0% | 一切なし |
| 障害対応 | 10% | Runbookなし。エスカレーションパスなし |
| バックアップ | 10% | DBバックアップ未確認 |
| デプロイ | 50% | CI/CDあり。カナリア/ロールバック/安全装置なし |
| シークレット管理 | 40% | 環境変数利用。Vault/ローテーション/スキャンなし |
| テスト | 40% | Jest設定済み。カバレッジ閾値なし。負荷テストなし |

### 最重要ギャップ

1. **DBバックアップが存在しない** — DB消失=全データ永久喪失
2. **アラート0件** — 500エラー急増、DB接続断、クレジット付与失敗→誰にも通知されない
3. **Pinoが入っているのに使われていない** — ログは全てコンソール出力→コンテナ再起動で消失
4. **スケジュールジョブの多重実行リスク** — 複数インスタンスで全て実行→クレジット二重付与
5. **ヘルスチェックが不完全** — PostgreSQLのみ。Redis/Meilisearch/Claude API未確認
6. **デプロイの安全装置なし** — CI失敗でもデプロイされる。ロールバック手順なし
7. **監査ログなし** — クレジット消費、ユーザーBAN、サブスク変更の操作記録なし
